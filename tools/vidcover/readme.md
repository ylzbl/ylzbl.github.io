# 视频封面生成器 (Video Cover Generator)

## 1. 项目简介

本项目是一个基于纯前端（HTML5 Canvas + Vanilla JavaScript）实现的**视频封面与遮罩生成工具**。

主要用途是帮助内容创作者快速生成统一视觉规范的视频封面（微信视频号、B站）、全屏透明水印遮罩，以及视频片尾的 Credits（演职人员）面板。

项目无需任何后端服务，所有图片合成、文字排版、缩放自适应、下载功能均在浏览器本地完成。

## 2. 主要功能模块

目前系统支持四种不同的生成模式：

1. **微信视频号封面 (1080x1448)**
   - 包含底图 (`bg.jpg`)、顶部 Logo (`logo.png`)、中部自定义封面图、底部紧凑组合的黄底黑字/黑底白字双行标题。
   - 标题具备字数检测和自动等比缩放功能。
2. **B站视频封面 (1920x1080)**
   - 包含底图、中部自定义封面图、底部紧凑组合双行标题。
   - 标题总高度限制在画面整体的1/3以内，左右边距自适应，超宽自动缩放。
3. **竖屏短标题 (透明底 1080x1920)**
   - 输出纯透明 PNG 图片，用于在视频剪辑软件中作为遮罩层叠加。
   - 画面正中心自动缩放放置水印图 (`watermark.png`)。
   - 顶部对齐放置两行标题。
4. **Credits竖屏 (1080x1920)**
   - 包含底图、顶部 Logo。
   - 支持多行自定义文字（上下两段，居中排版）。
   - 中部专门预留了“微信文章气泡截图”的上传位（默认加载 `search.jpg`），且渲染时自带右下角45度阴影立体效果。

## 3. 使用说明

### 3.1 环境准备

为保证最佳的本地加载体验，建议在 `index.html` 同级目录下放置以下缺省图片资源：

- `bg.jpg` (全局背景大图)
- `logo.png` (顶部品牌Logo)
- `watermark.png` (透明遮罩模式下的中心水印)
- `search.jpg` (Credits模式下默认的气泡截图)
- 字体文件：`AliPuHui-Regular.woff2` 及其他字重文件（如果有）。

### 3.2 界面交互

- **切换封面类型**：选择单选按钮即可实时切换预览画布的大小和排版。
- **上传图片**：支持点击上传或**拖拽图片**到虚线框内。Credits 模式和常规模式的图片上传状态是相互独立解耦的。
- **文本输入**：标题或正文输入时，右侧 Canvas 预览会实时更新。
- **本地记忆**：页面使用了 `localStorage`。您填写的文本、上传的小图、选择的模式会在刷新浏览器后自动恢复。
- **下载**：点击“下载封面图片”，即可将当前的 Canvas 渲染结果导出为 `.png` 图片保存到本地。

## 4. 最近更新修改记录 (Changelog)

- **修复与还原**：修复了“微信视频号封面”因为代码合并丢失的文字自适应缩放（`scale`）逻辑，确保长标题不会溢出画面。
- **排版重构**：将“B站视频封面”的左右并排文字，统一修改为与微信一致的上下组合紧凑排版，并加入了“不超过1/3高度”和“最大85%宽度”的智能缩放边界。
- **新增 Credits 模式**：新增 `1080x1920` 的片尾字幕模式，引入了 `<textarea>` 多行文本支持，以及带45度阴影的独立图片渲染逻辑。
- **新增竖屏透明模式**：竖屏短标题改为1080x1920透明全屏画布，自动计算中心水印位置及顶部文本位置。
- **容错与网络降级**：当本地图片资源缺失时，利用 `images.weserv.nl` 图像代理服务作为 Fallback 自动拉取网络备用图，并绑定 `onload` 触发重绘。
- **字体渲染修复**：增加对 `document.fonts.ready` 的监听，强制在字体库加载完毕后重绘，解决首次打开时阿里巴巴普惠体降级成微软雅黑的Bug。

## 5. 👨‍💻 开发者注意事项（二次开发与代码精简指南）

如果您准备对当前代码进行精简、重构或组件化（如拆分成 Vue/React），请务必注意以下**核心逻辑和暗坑**，不要在精简时误删：

### 1. Canvas 跨域污染 (Tainted Canvas) 问题

- **现象**：如果 Canvas 中绘制了跨域的图片，调用 `canvas.toDataURL()` 下载时会直接抛出安全错误（CORS 报错）。
- **避坑**：所有通过 JS `new Image()` 加载的网络图片，**必须**设置 `img.crossOrigin = 'anonymous'`。同时，图片所在的服务器（或 CDN）必须返回 `Access-Control-Allow-Origin: *` 响应头。
- *注：代码中使用 `images.weserv.nl/?url=...` 就是为了解决 GitHub Pages raw 图片缺少跨域头的问题。*

### 2. 字体异步加载陷阱

- **现象**：浏览器加载外部字体（`@font-face`）是异步的。如果图片、JS 先执行完调用了 `ctx.fillText()`，此时字体可能还没下载完，导致最终画出来的是默认字体（如 Arial/微软雅黑）。
- **避坑**：代码底部有 `document.fonts.ready.then(() => drawCanvas())`。如果您将代码移植到生命周期框架中，请确保在**字体加载就绪后**再执行最终渲染。

### 3. LocalStorage 存储配额限制

- **现象**：代码会将用户上传的图片转为 Base64 存储在 `localStorage` 中以便下次打开时恢复。但浏览器的 LocalStorage 上限通常只有 `5MB`。
- **避坑**：如果用户上传了极高分辨率的图片，转成 Base64 可能会超出配额导致 `QuotaExceededError`。
- 代码中的 `saveState()` 已经做了一层 `try...catch` 降级处理：如果超出配额，会抛弃图片数据，仅保存文本配置。**重构时切勿丢掉这个 catch 保护**。

### 4. 文本测算与中英文字符宽度缩放

- **逻辑**：代码中的 `countCharacters()` 函数，强制将汉字视为 `2` 个单位宽度，英文字母视为 `1` 个单位宽度。以此计算出一个 `effectiveWidth`。当计算宽度 > 18 时，触发等比缩小 `scale = 18 / maxWidth`。
- **避坑**：这是为了维持视觉效果统一的“伪排版引擎”。精简代码时，保留此函数以确保中英文混合输入时不会出现缩放抖动。

### 5. 图片的居中裁剪逻辑 (Object-Fit: Cover)

- 代码中的 `ctx.drawImage` 没有直接拉伸图片，而是使用了类似 CSS `object-fit: cover` 的算法计算 `imgAspect` 与 `areaAspect`。
- 在精简或抽象绘制图片的函数时，请保留按长宽比对比较后重新计算 `drawX`, `drawY`, `drawWidth`, `drawHeight` 的数学逻辑，防止用户上传的细长图或竖图被严重拉伸变形。

### 6. Credits 阴影状态未重置的污染

- 在 `drawCreditsCover` 中，绘制带有阴影的气泡截图时，使用了 `ctx.shadowColor` 等属性。
- **避坑**：Canvas 的状态机是全局共享的，如果不立即恢复 `ctx.shadowColor = 'transparent'`，阴影效果会“污染”后续绘制的所有文字，导致文字也带上黑影变得模糊。代码中已做恢复，重构抽取函数时请留意上下文状态（或使用 `ctx.save()` 和 `ctx.restore()` 包裹）。
